# 깃라잡이

초보 개발자가 오픈소스에 쉽게 기여할 수 있도록 돕는 플랫폼입니다.  
이슈 탐색, 스터디 관리, 온보딩 튜토리얼, 기여도 시각화, 커뮤니티 등을 통합하는 서비스 플랫폼

---

## 주요 기능
- **사용자 관리**: GitHub 로그인/로그아웃, 프로필 생성·편집, 회원 탈퇴
- **스터디 관리**: 스터디 생성·초대·탈퇴
- **이슈 탐색**: good first issue 검색
- **기여도/보상 시스템**: 내 기여 확인, 시각화, 랭킹, 뱃지, 도전과제
- **커뮤니티**: 스터디 모집, 게시판, 댓글

# 필독🚀Git 브랜치 및 Push 규칙
## 1. 핵심 브랜치 전략
* `main`: **(배포용)**
    * 실제 프로덕션 환경에 배포되는 가장 안정적인 코드가 있는 브랜치입니다.
    * **절대** 이 브랜치에 직접 커밋(Push)하지 않습니다.
    * 오직 릴리즈 준비가 완료된 `develop` 브랜치의 코드만 병합(Merge)됩니다.

* `develop`: **(통합용)**
    * 다음 배포 버전을 위해 개발이 완료된 기능들이 통합되는 메인 브랜치입니다.
    * **절대** 이 브랜치에 직접 커밋(Push)하지 않습니다.
    * 모든 기능은 `feature/*` 브랜치에서 **Pull Request (PR)**를 통해서만 병합됩니다.
 
## 2. 작업 브랜치 (Feature Branches)

* `feature/*`: **(기능 개발용)**
    * 새로운 기능을 개발할 때 사용하는 브랜치입니다.
    * **규칙:**
        1.  항상 최신 `develop` 브랜치에서 생성합니다.
        2.  브랜치 이름은 `feature/[기능]` 형식(소문자권장)을 따릅니다.
            * 예시 `feature/login`, `feature/user-profile`, `feature/post-crud`

## 3. 개발 워크플로우

**1단계: 기능 개발 시작**

1.  로컬 `develop` 브랜치를 최신 상태로 업데이트합니다.
    ```bash
    git checkout develop
    git pull origin develop
    ```
2.  새로운 기능 브랜치를 생성하고 이동합니다. (예: 로그인 기능)
    ```bash
    git checkout -b feature/login
    ```

**2단계: 개발 및 커밋**

1.  새로운 브랜치(`feature/login`)에서 코드를 작성하고 커밋합니다.
2.  커밋 메시지는 명확하게 작성합니다. (예: `feat: 깃허브 로그인 구현`)
3.  자신의 작업 브랜치는 원격(origin)에 자유롭게 푸시합니다.
    ```bash
    git push origin feature/login
    ```
